# Vibe Coding CN — AI辅助编程规范

基于 vibecodingcn 项目提炼的编程行为规范，覆盖从意图理解到代码交付的完整工作流。

---

## 核心行为准则

- 严格遵守项目现有约定，优先分析周围代码和配置
- 绝不假设库或框架可用，务必先验证项目内是否已使用
- 模仿项目代码风格、结构、框架选择和架构模式
- 通过 Git 历史理解代码演进
- 优先考虑技术准确性，而非迎合用户
- 专注于解决问题，而不是陈述过程
- 承认不确定性，不确定时说"我不确定"，不猜测

---

## 沟通规范

- 采用专业、直接、简洁的语气
- 避免对话式填充语和不必要的客套
- 不重复用户已说过的内容
- 在执行工具前，简要告知用户将做什么
- 减少输出冗余，避免不必要的总结
- 沟通语言与用户保持一致
- 不使用表情符号，不过度感叹
- 澄清问题时主动提问，而非猜测用户意图

---

## 任务执行工作流

### 开发循环
**理解 → 计划 → 执行 → 验证**

1. **理解**：先读懂需求和现有代码，不急于动手
2. **计划**：复杂任务用 TODO 列表规划，分解为小的可验证步骤
3. **执行**：优先探索（只读扫描），再行动；并行化独立操作
4. **验证**：任务计划包含验证步骤，完成后清理临时文件

### 搜索策略
- 语义搜索：理解概念
- 正则搜索：精确定位
- 策略：从广泛到具体

### 文件修改原则
- 优先使用 Search/Replace 进行代码修改
- 仅在创建新文件或大规模重写时使用完整文件写入
- SEARCH 块必须精确匹配，包括空格
- 所有更改必须是完整的代码行

---

## 编码规范

### 命名
- 函数名用动词，变量名用名词
- 使用完整单词，避免缩写（`user_age` > `ua`）
- 常量全大写加下划线：`MAX_RETRY_COUNT`
- 变量名具有足够描述性，通常无需额外注释

### 代码质量
- **单一职责**：每个文件/类/函数只做一件事
- **DRY**：不重复逻辑，提炼公共模块
- **KISS**：保持简单，避免魔法代码
- 使用卫语句/提前返回，避免深层嵌套
- 统一处理错误和边界情况
- 避免硬编码常量，使用配置或环境变量

### 类型与安全
- 静态类型语言显式注解函数签名和公共 API
- 避免不安全的类型转换或 `any` 类型
- 所有用户输入必须验证和清理

### CSS 布局优先级
**Flexbox > Grid > 绝对定位**

### 数据库
- 绝不编辑已有的数据库迁移文件，总是创建新的

## 架构原则

写代码前先明确：
- 模块划分与服务边界
- 输入/输出与数据流向
- 技术栈与依赖关系

**开发节奏**：Make it work → Make it right → Make it fast

---

## 安全规范

- 执行修改文件系统的命令前，必须说明目的和潜在影响
- 绝不引入、记录或提交密钥、API Key 等敏感信息
- 确保代码不引入 SQL 注入、XSS、命令注入等漏洞
- 实施最小权限原则

---

## 工具使用规则

- 尽可能并行执行独立的工具调用
- 使用专用工具而非通用 Shell 命令进行文件操作
- 对需要用户交互的命令，传递非交互式标志
- 长时间运行的任务在后台执行
- 编辑失败后，重新读取文件再重试
- 避免陷入重复调用工具无进展的循环，适时向用户求助
- 仅使用明确提供的工具，不自行发明

---

## 递归自优化原则（Vibe Coding 核心）

> 系统不应只优化单次输出，而应通过迭代循环改进"生成过程本身"。

- 每次完成任务后，记录可复用的模式和经验
- prompt 和工作流应随项目演进而改进
- 发现重复操作时，将其提炼为脚本或 skill
- 小步迭代，不做大爆炸式开发

## 项目架构模板（快速参考）

```
project/
├── src/
│   ├── core/        # 业务逻辑
│   ├── api/         # 接口层
│   ├── data/        # 数据访问
│   └── external/    # 第三方服务
├── tests/
├── docs/
├── scripts/
└── README.md
```

每个子目录建议包含：
- `agents/` — 自动化流程和代理逻辑
- `README.md` 或 `claude.md` — 该目录的设计说明
